# Proxmox SOC Lab — Pitfalls and Operational Notes

Comprehensive list of known pitfalls, bugs, workarounds, and session-specific operational notes for the Proxmox SOC Lab environment.

## General Infrastructure

- **vmbr2 already in use**: vmbr2 = 10.1.0.1/24 for techshop staging VMs (201-210). SOC Lab bridge is **vmbr3**. Never assign SOC Lab to vmbr2.
- **vmbr3 vs vmbr0 conflict**: vmbr0 carries existing student labs (SDN, vnets). SOC Lab uses *only* vmbr3. Never mix.
- **No IP on vmbr3**: Proxmox management is via vmbr0.10 (existing MGMT) or vmbr3.10 (10.10.10.2/24). OPNsense routes SOC VLANs.
- **VLAN parent bug**: When creating vmbr3.10 via API, `vlan-raw-device` may incorrectly show `vmbr2`. Verify with `cat /etc/network/interfaces` after creation.
- **Wazuh disk fills fast**: Set ILM before starting. Monitor with `df -h` and `_cat/indices`.
- **Snapshots ≠ backups**: Always `vzdump` to external storage. Snapshots are for pre-change rollback only.
- **RAM pressure**: Start VMs by module, not all at once. Total SOC lab is 42G. Coolify (701) uses 20G. Available ~39G — stop unused VMs before deploying.

## OPNsense

### Installation and Boot

- **OPNsense initial config requires VNC or OCR+sendkey**: QEMU guest agent is NOT pre-installed on OPNsense ISO. Console setup must be done via VNC. Steps: (1) Install to disk, (2) Verify partitions with `gpart show da0`, (3) Shutdown + remove ISO, (4) Boot, (5) Reassign interfaces every time after fresh install — OPNsense may invert vtnet0/vtnet1, (6) Set LAN IP 10.10.10.1/24, (7) Enable DHCP on LAN, (8) Access web UI via SSH tunnel (`ssh -L 10443:10.10.10.1:443 root@PVE_HOST`).
- **OPNsense ZFS install requires OVMF (UEFI)**: SeaBIOS cannot boot ZFS-formatted disks ("Booting from ROM... Non bootable"). With ZFS install, must use `bios=ovmf` + `efidisk0=local:1,efitype=4m,pre-enrolled-keys=0`. The earlier OVMF boot hang was with OPNsense 25.x — 26.1 with ZFS works on OVMF. UFS installs work fine with SeaBIOS. Always match bios to install format.
- **OPNsense 26.1 installer login**: Login as `installer`/`opnsense` (NOT root/opnsense). The boot menu no longer has "Install OPNsense" option — it shows: 1.boot multi user, 2.boot single user, 3.escape to loader, etc. Just boot multi user then login as installer.
- **OPNsense boot menu is FreeBSD loader, NOT GRUB**: With SeaBIOS, OPNsense 26.1 shows a FreeBSD loader menu (Boot multi user, Single user, Escape to loader, Reboot, etc.) — there is NO "Install OPNsense" entry. Press Enter/1 to boot into live mode, then install from the **OPNsense console menu**. Do NOT try automated GRUB sendkey sequences like "down arrow twice + enter" — those won't work.
- **OPNsense installer credentials**: Login as `installer` / `opnsense` (NOT root/opnsense) to launch the installer. root/opnsense is for the console after install.
- **OPNsense "select device to import from" prompt**: When booting from ISO with empty disk, OPNsense asks to import config from a device. Selecting `da0` fails ("invalid argument, /conf/config.xml not found"). **Leave blank and press Enter** to skip — this drops you to the console menu.
- **OPNsense installer login (26.1)**: The installer is NOT launched from the root console menu. You must login with user **`installer`** and password **`opnsense`** to launch the installer. The `root`/`opnsense` login drops you into live mode with the config menu — no install option there. The `opnsense-install` command does NOT exist in 26.1 (returns "command not found").
- **ZFS partition layout verification**: After ZFS install, `gpart show da0` should show: `efi` (260M), `freebsd-zfs` (~12G), `freebsd-swap` (8G). If only one tiny partition or nothing, the install failed silently — reinstall before rebooting.
- **ISO left mounted after OPNsense install**: After OPNsense is installed to disk, the ISO (`ide2`) stays attached with `boot=cdn`. Detach via API: `--data-urlencode "ide2=none,media=cdrom"` + `--data-urlencode "boot=dcn"`. Leaving the ISO causes re-entry into the installer on reboot.
- **OPNsense bios=firmware must match filesystem format**:
  - **UFS install + SeaBIOS** ✅ — Works. UFS creates an MBR boot sector that SeaBIOS can boot.
  - **ZFS install + SeaBIOS** ❌ — ZFS install creates a GPT/EFI partition (260MB). SeaBIOS cannot boot from EFI partitions. Result: "Booting from ROM... Non bootable" with 0 useful disk activity and ~36-37MB RAM usage (no OS loaded). **Fix**: either reinstall with UFS format, or switch to OVMF (`bios=ovmf`) + add `efidisk0`. For a lab, UFS is simpler and more reliable.
  - **OVMF + ZFS works in 26.1**: OVMF boot hang was a 25.x bug. In 26.1, OVMF+q35+ZFS boots correctly (confirmed: 1.7GB RAM usage, OPNsense fully operational). Only use SeaBIOS if doing UFS install.
- **Diagnostic via API**: After booting from disk, check RAM via `status/current`. ~36MB = no OS loaded (empty disk or wrong boot method). ~1.3GB+ = OPNsense running normally.

### Networking

- **Hetzner MAC filtering (CRITICAL)**: On Hetzner, vmbr0 bridges to eno1 physical NIC. Hetzner switches only allow the host's real MAC. OPNsense vtnet0/e1000 MAC (BC:24:11:XX) will be **BLOCKED** — no WAN DHCP, no internet, ping returns "host unreachable". **Fix**: Do NOT assign public IP to OPNsense WAN. Instead, NAT via PVE host: private /24 link on vmbr0 (10.0.0.1/24 on PVE, 10.0.0.2/24 on OPNsense WAN), plus PVE host iptables MASQUERADE + FORWARD rules. OPNsense WAN gateway = 10.0.0.1 (PVE host), NOT the public upstream gateway. See `references/hetzner-nat-wan.md` for full two-level NAT setup. **Missing PVE host NAT is the #1 cause of "ping gateway works but ping internet fails"** — iptables rules must be added AND persisted in `/etc/network/interfaces`.
- **OPNsense WAN is STATIC with private IP behind PVE NAT (Hetzner)**: OPNsense WAN = `10.0.0.2/24` on vmbr0, PVE host = `10.0.0.1/24` on vmbr0. **WAN gateway MUST be 10.0.0.1** (PVE host) — NOT the public upstream gateway (95.217.87.65). Setting the public gateway causes "Destination Host Unreachable" because OPNsense can't ARP for 95.217.87.65 on the 10.0.0.0/24 subnet. PVE host does the NAT to reach the public internet. **Two-level NAT required**: (1) OPNsense masquerades 10.10.x.x → 10.0.0.2, (2) PVE host masquerades 10.0.0.0/24 → 95.217.87.114 (public). Both levels MUST be configured. Missing PVE NAT is the #1 cause of "VMs can ping gateway but not internet" — see `references/hetzner-nat-wan.md` for full setup.
- **OPNsense DNS SERVFAIL after fresh install**: Even with NAT outbound (Hybrid), Unbound may fail to forward. Checklist: (1) System→Settings→General: add DNS 8.8.8.8+1.1.1.1, (2) Services→Unbound→General: Enable + check Listen Interfaces, (3) Firewall→Rules→WAN: ensure outbound DNS not blocked, (4) Test from console: `host deb.debian.org 8.8.8.8` and `host deb.debian.org 127.0.0.1` — if OPNsense itself can't resolve, issue is WAN-side.
- **OPNsense DNS/NAT not configured by default**: After initial install, OPNsense does NOT forward DNS or NAT LAN traffic to WAN by default. **Outbound NAT is mandatory for internet access** — without it, VLAN clients can ping their gateway (10.10.X.1) and even the WAN gateway (10.0.0.1), but traceroute stops there. Configure via Firewall → NAT → Outbound → select **Automatic** or **Hybrid** mode → Save → Apply. With Automatic mode, OPNsense creates masquerade rules for all internal networks. Without this, source IPs (10.10.X.X) are not translated and are dropped by the upstream router. After enabling, verify with `ping 8.8.8.8` from any VLAN VM.
- **OPNsense DNS tools**: `nslookup` does NOT exist on OPNsense. Use `host <domain> <server>` or `drill <domain> @<server>` instead.
- **OPNsense web UI SSH tunnel from PVE host**: Cannot reach OPNsense web UI from outside without tunnel. From local PC: `ssh -L 10443:10.10.10.1:443 root@100.70.37.62`, then browse `https://localhost:10443`. Do NOT open port 443 on the Hetzner firewall — use SSH tunnel only.
- **OPNsense interface inversion**: After install, OPNsense may assign vtnet0 as LAN and vtnet1 as WAN — the opposite of the intended setup (vtnet0=vmbr0=WAN, vtnet1=vmbr3=LAN trunk). Always verify via console option `1` (Assign interfaces) right after first boot. Fix: reassign WAN=vtnet0, LAN=vtnet1_vlan10 with VLANs 10-70 on parent vtnet1.
- **OPNsense Unbound DNS**: Must explicitly add LAN interface to **Listen Interfaces** (Services → Unbound DNS → General). Without this, Unbound only listens on localhost. Also add DNS forwarders (8.8.8.8, 1.1.1.1) in System → Settings → General. Kea DHCP subnets should have DNS=OPNsense VLAN IP as DNS server.
- **OPNsense version**: Current stable is **26.1.x** (not 25.1). ISO at `https://pkg.opnsense.org/releases/26.1/OPNsense-26.1.2-dvd-amd64.iso.bz2`. `mirror.opnsense.org` DNS often fails — use `pkg.opnsense.org`.
- **OPNsense requires 3 NICs for VLAN trunking**: OPNsense needs: net0=vmbr0 (WAN), net1=vmbr3,tag=10 (MGMT VLAN, untagged inside OPNsense), **net2=vmbr3 (trunk, no tag)**. Without net2 as trunk, the trunk NIC only carries untagged traffic. After adding net2, OPNsense creates sub-interfaces on the trunk NIC.
- **⚠️ CRITICAL: OPNsense trunk NIC MUST be `e1000`, NOT `virtio`**: The FreeBSD virtio driver does NOT properly handle VLAN-tagged frames on trunk (parent) interfaces. Symptom: VMs on vmbr3 tag=X send ARP for gateway but get INCOMPLETE — the trunk NIC silently drops VLAN-tagged frames. Fix: change trunk NIC model from `virtio` to `e1000`. **Access port VMs (individual VLAN tag=X) can stay `virtio`** — PVE strips/adds the VLAN tag at the bridge level, so the VM NIC never sees tagged frames. Only the trunk NIC (OPNsense net2 carrying all VLANs) needs e1000.
- **OPNsense trunk NIC model change requires VLAN recreation**: Changing a NIC from `virtio` to `e1000` changes the FreeBSD interface name (e.g., `vtnet1` → `em2`). All VLANs configured on `vtnet1_vlanX` become orphaned — they still appear in OPNsense but point to a non-existent parent. **Must delete old VLANs and recreate on new parent** (e.g., delete `vtnet1_vlan20`-`vtnet1_vlan70`, recreate as `em2_vlan20`-`em2_vlan70`). Reassign opt interfaces to the new VLAN sub-interfaces. This must be done via OPNsense web UI or VNC console — no REST API for VLAN management.
- **OPNsense current NIC layout (after e1000 migration)**: net0=e1000 (WAN→em0), net1=e1000 (MGMT→em1, vmbr3 tag=10), net2=e1000 (trunk→em2, vmbr3 no tag). **ALL 3 NICs must be e1000** — not just the trunk. VLANs are `em2_vlanX` (NOT `vtnet1_vlanX`). Note: opt numbers shifted after VLAN recreation — was opt4=DMZ, now opt3=DMZ.
- **OPNsense VLAN status (May 2026, after e1000 migration)**: All VLANs recreated on `em2` parent. em0=WAN (10.0.0.2/24), em1=LAN/MGMT (10.10.10.1/24), em2=trunk (no IP, must stay enabled). VLANs: em2_vlan20=opt1/SOC (10.10.20.1/24), em2_vlan30=opt2/IT (10.10.30.1/24), em2_vlan40=opt3/DMZ (10.10.40.1/24), em2_vlan50=opt4/OT (10.10.50.1/24), em2_vlan60=opt5/Suricata (10.10.60.1/24), em2_vlan70=opt6/Quarantine (10.10.70.1/24). **All /24 confirmed** (some originally /32, fixed via web UI). DHCP configured on all VLANs (range .100-.200). Firewall rules: AllowAll on all VLANs (temporary).
- **OPNsense trunk interface (opt7 or equivalent) MUST stay ENABLED with NO IP**: When OPNsense has net2 as trunk (vmbr3, no tag), it appears as an opt interface (e.g., `opt7/NET2` with `em2` device after e1000 migration). **This interface MUST remain enabled** (Interfaces → Assignments → click the trunk opt → Enable = checked) but with NO IP address. If disabled, VLAN sub-interfaces lose their parent and cannot send/receive tagged frames. **Confirmed**: Disabling trunk opt causes VLAN ARP INCOMPLETE on all sub-interfaces. With e1000 NICs, the trunk device is `em2` (not `vtnet1`).
- **VLAN connectivity debugging — ARP INCOMPLETE pattern**: When a VM on vmbr3 tag=X sends ARP for the gateway (10.10.X.1) and gets INCOMPLETE, the problem is NOT in the VM config. The issue is OPNsense-side. Debug in order: (1) **Is the trunk NIC `e1000`?** FreeBSD virtio driver silently drops VLAN-tagged frames on trunk — this is the #1 cause. Change to e1000 and recreate VLANs. (2) Check that the trunk opt interface is enabled (no IP needed, but must be enabled), (3) Verify VLAN sub-interface has `status: up` and correct IP in `interfacesInfo` API, (4) Verify firewall rules exist on the interface AND are actually applied, (5) Verify in OPNsense web UI: Interfaces → VLANs → parent matches the trunk NIC name (e.g., `em2` after e1000 migration), (6) After ANY OPNsense config change, reboot OPNsense. **Key lesson**: virtio trunk = ARP INCOMPLETE, e1000 trunk = VLAN works. Access-port VMs can stay virtio.
- **OPNsense SNAT (Source NAT) is required for VLAN internet access**: OPNsense needs Source NAT rules to translate VLAN source IPs (10.10.X.X) to the WAN IP (10.0.0.2) so that return traffic can be routed back. Without SNAT, VLAN traffic reaches the PVE host with private source IPs that the PVE host can't route back. **Configure via OPNsense API**: `POST /api/firewall/filter/addRule` for each VLAN with `snat=1`, `target=wan_ip`, `interface=wan`, `from=10.10.X.0/24`. Then `POST /api/firewall/filter/apply`. Verify from VLAN VM with `curl -sI https://google.com`.
- **OPNsense internet access requires ALL 4 of: PVE host NAT + WAN gateway + outbound NAT + DNS forwarding**: After initial install, no VLAN client can reach internet. The complete checklist:
  1. **PVE host NAT/masquerade (CRITICAL, NOT automatic)**: The PVE host must NAT traffic from 10.0.0.0/24 → public IP. Without this, OPNsense can reach 10.0.0.1 but traffic dies there — PVE host has no route back to 10.10.x.x. Run on PVE host: `iptables -t nat -A POSTROUTING -s 10.0.0.0/24 -o eno1 -j MASQUERADE` + `iptables -A FORWARD -s 10.0.0.0/24 -o eno1 -j ACCEPT` + `iptables -A FORWARD -d 10.0.0.0/24 -o vmbr0 -m state --state RELATED,ESTABLISHED -j ACCEPT`. **Persist in `/etc/network/interfaces`** under vmbr0 stanza with `post-up` lines (see `references/hetzner-nat-wan.md`). Symptom of missing PVE NAT: `traceroute 8.8.8.8` from any VM shows hop 1 = OPNsense (10.10.X.1), then * * * (packets dropped at PVE host).
  2. **WAN gateway must be 10.0.0.1 (PVE host), NOT the public upstream gateway**: OPNsense WAN (10.0.0.2/24) shares a private /24 with PVE host (10.0.0.1). The next hop MUST be 10.0.0.1. Setting the gateway to the public upstream gateway (e.g., 95.217.87.65) **WILL NOT WORK** — OPNsense cannot ARP for 95.217.87.65 on the 10.0.0.0/24 subnet. PVE host does the NAT to reach 95.217.87.65. Configure: System → Routes → Gateways → WAN_GW → 10.0.0.1, then Interfaces → WAN → IPv4 Upstream Gateway = WAN_GW.
  3. **OPNsense outbound NAT (masquerade)**: Firewall → NAT → Outbound → select **Automatic** or **Hybrid** mode → Save → Apply. This translates 10.10.x.x → 10.0.0.2 (OPNsense WAN IP) before sending to PVE host. Without this, OPNsense routes VLAN traffic with source 10.10.x.x which PVE host doesn't know how to route back. Symptom: VMs ping gateway but traceroute dies at 10.0.0.1.
  4. **DNS forwarding**: Services → Unbound DNS → General → Enable + add LAN interfaces to Listen Interfaces. System → Settings → General → add DNS servers (8.8.8.8, 1.1.1.1). Kea DHCP subnets must have DNS=OPNsense VLAN IP (e.g., 10.10.10.1 for MGMT, 10.10.20.1 for SOC, etc.) as DNS server so clients can resolve names. Without DNS server in DHCP config, clients get no DNS — `ping 8.8.8.8` works but `apt update` fails.
  **Two-level NAT architecture**: OPNsense masquerades 10.10.x.x → 10.0.0.2 (WAN IP), then PVE host masquerades 10.0.0.x → 95.217.87.114 (public IP). Both levels MUST be configured for internet to work. Verify from any VLAN VM: `ping 8.8.8.8` (both NAT levels), `host deb.debian.org` (DNS test).
- **VLAN must be ACTIVE on OPNsense before migrating VMs**: Before moving a VM from vmbr1 to vmbr3 tag=X, confirm that OPNsense has the VLAN X interface UP with an IP (10.10.X.1/24) and DHCP pool configured. Without this, the VM will have no gateway, no DNS, and no connectivity. Test: `ping 10.10.X.1` from PVE host (on vmbr3.10) or from the VM after applying the IP manually. If ping fails, OPNsense VLAN X is not configured or not enabled — do NOT migrate that VM until the VLAN is active.

### Authentication and API

- **OPNsense web UI auth requires CSRF token**: Login endpoint is `POST /index.php?login` (NOT `/api/` endpoints). Flow: (1) `GET /` → extract CSRF token from HTML, (2) `POST /index.php?login` with `csrf_token=<token>&username=<user>&password=<pass>` + session cookie. API endpoints under `/api/` require API keys (System → Access → API Keys) — HTTP Basic auth with root/password returns 401. Default `root:opnsense` does NOT work after initial setup.
- **OPNsense password reset requires VNC console**: If the OPNsense web UI password is lost, the ONLY way to reset it is via VNC console (PVE web UI → VM 300 → Console). At the OPNsense menu: **option 3** ("Reset the login password") or **option 8** (Shell) → `passwd root`. There is NO remote API to reset credentials. The QEMU monitor `sendkey` API can send keystrokes but is write-only (cannot read console output) — useless for interactive password reset.
- **OPNsense web UI login (CSRF flow)**: Login endpoint is `POST /login` (NOT `/api/core/auth/login` which returns 404). Flow: (1) `GET /` → extract CSRF token from JS (`setRequestHeader("X-CSRFToken", "<token>")`), (2) Extract hidden form field name+value (dynamic CSRF field that changes each request, e.g. `name="YhR7TGtL..." value="Fy9YeisX..."`), (3) `POST /login` with `<hidden_field_name>=<hidden_value>`, `usernamefld=<user>`, `passwordfld=<pass>`, `login=Login`, `X-CSRFToken: <token>` header, and session cookie. **Field names are `usernamefld` and `passwordfld`** (NOT `username`/`password`). **Important**: curl-based login attempts often fail silently (redirect to `/?url=/login` with title "Se connecter | OPNsense") even with correct credentials due to cookie handling nuances. Python `requests.Session()` with proper cookie jar is more reliable. A successful login redirects to `/` (dashboard, title="OPNsense Dashboard"), failed login redirects to `/?url=/login` (title="Se connecter | OPNsense"). API endpoints under `/api/` require separate API keys (System → Access → API Keys) — HTTP Basic auth with `root:opnsense` returns 401 after initial setup.
- **OPNsense default credentials DO NOT survive initial setup**: `root:opnsense` only works on the first live-boot console. After running the installer and setting a new password (or after importing a config), the default is gone. The web UI uses the same `root` account as the console, so if console login works but web UI fails, the password has been changed. If the password is unknown, reset via OPNsense console: boot → option 3 ("Reset the login password") or option 8 (Shell) → `passwd root`. No remote reset possible — requires VNC or serial console access.
- **OPNsense API auth**: The `/api/` endpoints require API keys created in System → Access → APIKeys within the OPNsense web UI. HTTP Basic auth with root/password is NOT supported for `/api/` endpoints (returns 401). API keys are generated as `key:secret` pairs and used with `curl -u "$KEY:$SECRET"` or Python `requests.Session(auth=(KEY, SECRET))`. Without web UI access and valid credentials, the API is completely inaccessible. Create API keys via: System → Access → Users → root → API keys → ➕ (plus icon). Copy both key and secret immediately — secret is shown only once.
- **OPNsense API key permissions**: API keys have full admin access. They authenticate for `/api/*` REST endpoints but do NOT provide web UI session access (PHP pages like `/diag_backup.php`, `/interfaces.php` redirect to login). For configuration tasks not covered by the REST API, you must use the web UI via SSH tunnel (`ssh -L 10443:10.10.10.1:443 root@100.70.37.62`).
- **OPNsense API key creation**: Create via web UI: System → Access → Users → root → API keys → ➕. Key format: `key/secret` pair used as HTTP Basic auth (`curl -u "$KEY:$SECRET"`). Keys provide full admin access to `/api/*` endpoints but do NOT grant web UI session access (PHP pages redirect to login). Secret is shown only once at creation — save immediately.
- **OPNsense API key creation via SSH console**: From OPNsense SSH shell, run `opnsense-apikey -k <name> -s <secret>` to create API keys without web UI access. Useful when web UI is unreachable or for automation. The key and secret are printed on stdout. SSH access: `sshpass -p 'opnsense' ssh -o StrictHostKeyChecking=no -o PubkeyAuthentication=no root@10.10.10.1` (password auth forced since key auth fails after retries). NOTE: Shell variable expansion ($()) breaks inside double-SSH — pass literal values for `-s`.
- **OPNsense API keys saved in `/opt/data/.env`**: OPNsense API key and secret are stored as `OPNSENSE_API_KEY` and `OPNSENSE_API_SECRET` in `/opt/data/.env`. Always check these env vars before asking the user for API credentials. Use HTTP Basic auth with the key:secret pair for all `/api/*` endpoints.
- **OPNsense root SSH password**: The default password `root:opnsense` still works on SSH (as of May 2026). Access from PVE host: `sshpass -p 'opnsense' ssh -o PubkeyAuthentication=no root@10.10.10.1`. OPNsense SSH rejects key auth after too many attempts — always use `-o PubkeyAuthentication=no` with password auth.
- **OPNsense API key password reset**: If API key credentials are lost or OPNsense is rebooted and the key becomes invalid, create a new key via VNC console → option 8 (Shell) → `opnsense-apikey -k <key> -s <secret>` (or regenerate via web UI). The API key is tied to the root user account.
- **OPNsense API key lost after reboot**: OPNsense API keys persist across reboots (they are stored in config.xml). If authentication fails after reboot, the key string may have been mistyped or URL-encoding may differ. Use `-u "key:secret"` with curl (the key contains `/` which may need careful quoting). Verify the key works with `GET /api/core/system/status` before proceeding. If the key is truly lost, recreate via VNC console → option 8 (Shell) → `opnsense-apikey` or web UI System → Access → Users → root → API keys.
- **OPNsense reboot can break web UI access**: Rebooting OPNsense via API (`POST /api/core/system/reboot`) may invalidate the web GUI session or reset network config if VLAN interfaces aren't saved to config.xml. After reboot, always verify: (1) OPNsense responds on 10.10.10.1:443, (2) VLANs still assigned and enabled in Interfaces → Assignments, (3) API key still works. If unreachable, access via VNC console (PVE web UI → VM 300 → Console). If web UI becomes unresponsive after reboot, try `service_reload_all` via API or reload all services from VNC console option 11.
- **OPNsense API firewall rules unreliable — always verify via web UI**: Rules pushed via `POST /api/firewall/filter/set` + `POST /api/firewall/filter/apply` may NOT take effect even when API returns `{"result":"saved"}` and `{"status":"OK"}`. If traffic doesn't flow after pushing rules, (1) log into web UI Firewall → Rules → check rules exist and are enabled, (2) click orange "Apply Changes" button, (3) try `POST /api/core/service/restart/filter`, (4) last resort: reboot OPNsense. The API apply endpoint does NOT reliably reload the PF ruleset.
- **OPNsense API `/api/interfaces/settings/set` may silently fail on subnet mask**: Setting an interface IP via API with `subnet=24` may only save the IP address but not the subnet (interface ends up as /32). This was observed on multiple VLAN interfaces after bulk configuration. **Always verify** the subnet after API calls by checking `GET /api/interfaces/overview/interfacesInfo` — look for the `addr4` field to show the full CIDR (e.g., `10.10.30.1/24`, NOT `10.10.30.1/32`). If /32 appears, fix via OPNsense web UI (Interfaces → click interface → save with correct /24).
- **OPNsense lacks QEMU guest agent**: VM 300 (OPNsense) has `agent: 0` (not configured). Cannot use PVE agent/exec API for OPNsense. Must access OPNsense via VNC console (PVE web UI) or SSH tunnel + web UI. The QEMU monitor `sendkey` API works (`PUT /nodes/$NODE/qemu/300/monitor` with `command=sendkey X`) but there's no way to read console output via API — it's write-only keystrokes.

### VLAN Configuration (Hybrid API + Web UI)

- **OPNsense VLAN configuration — hybrid API + web UI workflow**: The REST API can partially configure VLANs but NOT fully activate them. The `os-interfaces` and `os-firewall` plugins were installed via `/api/core/firmware/install/os-interfaces` and `/api/core/firmware/install/os-firewall` — these add endpoints ONLY after OPNsense is rebooted. Available API endpoints after reboot include `/api/interfaces/overview/interfacesInfo`, `/api/interfaces/settings/get|set`, `/api/firewall/filter/get|set|apply`, `/api/core/service/search|restart|start|stop/<service>`. **NOT available via API**: Interface assignment (VLAN → optX), DHCP pool configuration, interface activation/enable toggle. These MUST be done via web UI.
- Configuration workflow per VLAN: (1) Interfaces → Assignments → add VLAN interface, (2) click new interface → enable + set static IP (10.10.X.1/24), (3) Services → Kea DHCPv4 → add subnet (10.10.X.100-10.10.X.200, DNS=gateway IP), (4) Firewall → Rules → [interface] → add allow-all rule temporarily. The API `/api/interfaces/settings/set` can pre-save IP configs but they only take effect after the web UI assignment step.
- **API-saved firewall rules may not take effect immediately**: Even after `/api/firewall/filter/apply` returns OK, rules may not be active. Verify in web UI Firewall → Rules that rules appear and are enabled. ## DNAT, NAT, and Port Forwarding

- **⚠️ CRITICAL: PVE PREROUTING DNAT rules MUST specify destination IP**: The PVE host has DNAT rules in PREROUTING to forward public IP ports (80, 443, 8443) to internal VMs (Coolify, Wazuh). These rules MUST match on `-d 95.217.87.114` (the PVE public IP), NOT `-d 0.0.0.0/0`. Rules matching `0.0.0.0/0` intercept ALL outbound TCP from OPNsense/VLANs. **Symptom**: `ping 8.8.8.8` works but `curl https://google.com` times out. Conntrack shows `SYN_SENT [UNREPLIED]` with incorrect reverse mapping. **Fix**: Change PREROUTING DNAT rules from `-d 0.0.0.0/0` to `-d 95.217.87.114`. After fixing, persist with `iptables-save > /etc/iptables/rules.v4` and flush conntrack with `conntrack -F`.
- **PVE host iptables NAT rules are NOT persisted across reboots**: The `iptables -t nat -A POSTROUTING` rules and `iptables -A FORWARD` rules added on the PVE host to enable internet for OPNsense VLANs are lost on reboot. **MUST persist** via: (1) `iptables-persistent` package + `iptables-save > /etc/iptables/rules.v4` for base rules, (2) `/etc/sysctl.d/99-soc-lab.conf` for sysctl, (3) **systemd service `soc-lab-iptables.service`** that runs AFTER `tailscaled.service` with `ExecStartPre=/bin/sleep 5` to re-add ts-postrouting RETURN rules, DNAT/SNAT for Wazuh, policy routing ip rules, and bridge-nf sysctl. Tailscale recreates its `ts-postrouting` chain on every restart, wiping any RETURN rules — the systemd service restores them. See `references/hetzner-nat-wan.md` for the exact configuration.
- **Conntrack debugging for DNAT issues**: When TCP connections fail but ICMP works, use `conntrack -L` on the PVE host to inspect connection tracking entries. Look for `SYN_SENT [UNREPLIED]` entries. The `conntrack -F` command flushes all entries and forces re-evaluation — essential after fixing iptables rules.
- **⚠️ Tailscale on PVE host breaks NAT for forwarded traffic (CRITICAL)**: When Tailscale is installed on the PVE host, its netfilter hooks (`ts-postrouting` chain with `mark match 0x40000/0xff0000` MASQUERADE) intercept and overwrite SNAT/MASQUERADE rules for traffic forwarded from OPNsense WAN and VLAN subnets. **Symptom**: Conntrack shows reply source as `10.0.100.2` (Tailscale IP) instead of `95.217.87.114` (public IP), making TCP connections fail while ICMP works. **Workaround**: Use PVE host VLAN interfaces (10.10.X.2) as direct gateways for VMs, bypassing OPNsense entirely. **If OPNsense WAN internet is needed**: The only reliable fix is to either (a) add a separate routing table that bypasses Tailscale, (b) add iptables rules that clear the Tailscale fwmark for forwarded packets before ts-postrouting, or (c) configure Tailscale with `--netfilter-mode=off` or `--netfilter-nat-rules=false` to prevent it from managing iptables NAT rules. Option (c) is the cleanest long-term fix but requires testing.
- **PVE gateway bypass pattern for VLAN VMs**: When OPNsense outbound TCP is blocked, use PVE host VLAN interfaces as direct gateways. Each VLAN has a PVE host IP: vmbr3.10=10.10.10.2, vmbr3.20=10.10.20.2, etc. These already have MASQUERADE rules. To bypass: (1) `ip route replace default via 10.10.X.2` on the VM, (2) fix DNS to 8.8.8.8 (disable Tailscale DNS override), (3) persist in `/etc/network/interfaces`. This is the RECOMMENDED approach for lab VMs that need internet access.
- **PVE host VLAN interfaces serve as bypass gateways**: The PVE host has `vmbr3.10` (10.10.10.2/24), `vmbr3.20` (10.10.20.2/24), etc. These provide L3 connectivity to all VLANs. Existing MASQUERADE rules NAT traffic from any VLAN to the internet via the PVE host. When OPNsense firewall rules block outbound traffic, use the PVE host VLAN IP as gateway to bypass.
- **⚠️ `bridge-nf-call-iptables` MUST be 0 for DNAT/port-forwarding to work (CRITICAL)**: When `net.bridge.bridge-nf-call-iptables=1`, bridged packets on vmbr3 traverse iptables chains a second time with the bridge's output interface instead of the VLAN sub-interface. This causes DNAT + MASQUERADE/SNAT rules to fail. **Result**: TCP connections time out — SYN reaches Wazuh but SYN-ACK never returns to the client. ICMP (ping) works because it doesn't need NAT reverse. **Fix**: `sysctl -w net.bridge.bridge-nf-call-iptables=0`. **Persist** in `/etc/sysctl.d/99-bridge-nf.conf`.
- **DNAT + SNAT pattern for exposing internal VM services on the public IP**: To expose a VLAN VM service (e.g., Wazuh UI on 10.10.20.10:443) on the PVE public IP, you need THREE rules: (1) **DNAT** in PREROUTING: `iptables -t nat -A PREROUTING -d 95.217.87.114 -p tcp --dport 8443 -j DNAT --to-destination 10.10.20.10:443`, (2) **SNAT** in POSTROUTING: `iptables -t nat -A POSTROUTING -d 10.10.20.10 -j SNAT --to-source 10.10.20.2` — this makes Wazuh see the request from its gateway, (3) **INPUT rule**: `iptables -I INPUT -p tcp --dport 8443 -j ACCEPT`. - **Wazuh UI external access URL**: `https://95.217.87.114:8443` — DNAT from PVE public IP port 8443 → Wazuh 10.10.20.10:443. Requires `bridge-nf-call-iptables=0` + DNAT + SNAT rules.

Stopping `tailscaled` (even temporarily) severs the SSH connection if the user connects via Tailscale. If Tailscale must be restarted, do it in a single command: `systemctl restart tailscaled`. If you must stop Tailscale, ensure `95.217.87.114` (public IP) is reachable and SSH works via that IP first.

## SSH Access Patterns

- **SSH to PVE host when sshpass is unavailable**: The Hermes container lacks `sshpass`, `paramiko`, `pip`, and `sudo`. ## OpenPLC v4

- **OpenPLC v3 is EOL**: Use **OpenPLC v4** (`autonomy-logic/openplc-runtime`) instead. Install: `git clone https://github.com/autonomy-logic/openplc-runtime.git && cd openplc-runtime && ./install.sh`. Creates systemd service on port 8443.
- **OpenPLC v4 SAFE MODE pitfall**: The runtime process is launched with `--safe-mode` flag when no compiled program exists. Even after successfully uploading and compiling a program, the running process continues in SAFE MODE. The REST API `start-plc` call does NOT exit safe-mode. **Fix**: `systemctl restart openplc-runtime`. If still in SAFE MODE after restart, the `.so` is missing or broken.
- **OpenPLC v4 REST API auth**: Create user first: `POST /api/create-user` with `{"username":"admin","password":"..."}`. Then login: `POST /api/login` returns JWT. Use `Authorization: Bearer ***` for subsequent requests.
- **OpenPLC v4 program upload format**: The ZIP must contain C source files. Minimum required: `Config0.c`, `Res0.c`, `debug.c`, `glueVars.c`, `c_blocks_code.cpp`, `program.st`, `lib/iec_types.h`, and optionally `conf/` for plugin configs.
- **OpenPLC v4 Modbus Slave plugin**: Must be enabled via config files. Create `conf/modbus_slave_config.json` with `{"slave_id":1, "port":502, "enable":true}` in the upload ZIP.
- **OpenPLC v4 .so symbol names (CRITICAL)**: The compiled `libplc_*.so` must export symbols with EXACT names expected by `plc_main`'s `dlsym()` calls: `config_run__`, `config_init__`, `glueVars`, `glueVars_init`, `setBufferPointers`, `setBufferPointers_v4`, `get_var_count`, `updateTime`, `plc_program_md5`, `set_endianness`. The default names generated by compile.sh do NOT match. You must rewrite the generated C files with the correct function names. Verify with: `nm -D build/libplc_*.so | grep -E 'config_run__|config_init__|...'`.
- **OpenPLC v4 compile.sh outputs `new_libplc.so`**: The `scripts/compile.sh` compiles to `build/new_libplc.so`, but `plc_main` loads `build/libplc_*.so` (glob pattern). After running compile.sh, you MUST: (1) remove old `.so` files, (2) rename `new_libplc.so` to `libplc_<timestamp>.so`. Without this rename, `plc_main` finds no program and stays in safe-mode.
- **OpenPLC v4 API is HTTPS, not HTTP**: The Flask webserver on port 8443 uses HTTPS. All API calls must use `https://`.
- **OpenPLC v4 `start-plc` API does NOT exit safe-mode**: Calling `GET /api/start-plc` just restarts `plc_main` but keeps the `--safe-mode` flag. The root cause is always a missing/broken `.so`. Fix the `.so`, then `systemctl restart openplc-runtime`.
- **OpenPLC v4 Modbus TCP 502 is served by plc_main (C), not Python plugin**: Even if the Python `modbus_slave` plugin fails, the native Modbus server in `plc_main` handles port 502 once the PLC is running with a valid program. Verify with `ss -tlnp | grep 502`.
- **OpenPLC v4 startup sequence**: `start_openplc.sh` launches Flask, which spawns `./build/plc_main`. If `plc_main` was previously in SAFE MODE, it stays in SAFE MODE even after `systemctl restart`. Check `ps aux | grep plc_main` for `--safe-mode` flag.

## ScadaBR

- **ScadaBR 1.2 needs Tomcat 9, not 10**: ScadaBR 1.2 uses `javax.*` namespace (Java EE) and fails with `Context startup failed` on Tomcat 10 (Jakarta EE). Install Tomcat 9 manually from Apache archives to `/opt/tomcat9/`. Fix: `db.type=mysql` in `env.properties`, and patch `createTables-mysql.sql` changing BOTH `script varchar(16384)` → `script TEXT` AND `xmlConfig varchar(16384)` → `xmlConfig TEXT`.
- **ScadaBR config via DWR + Selenium**: ScadaBR 1.2 has no REST API. DWR calls can automate datasource creation, but **DWR2 Object_Object:{} format ignores enum properties** — they default to 1 (Binary/Coil). **Workaround**: Use Selenium + headless Chromium to manipulate HTML selects directly. See `references/scadabr-dwr-automation.md`.
- **ScadaBR DWR session flow (MANDATORY)**: Before ANY save call, you MUST: (1) Login via POST `/ScadaBR/login.htm`, (2) GET `data_source_edit.shtm?typeId=X` (new DS) or `data_source_edit.shtm?dsid=X` (existing), (3) POST `DataSourceEditDwr/editInit` — this puts the DS object into the HTTP session. Skipping step 2→3 causes NullPointerException on every save call.
- **ScadaBR env.properties gets overwritten**: When Tomcat deploys/undeploys the WAR, it re-extracts `env.properties` from the WAR, overwriting any changes. To persist the DB config, either: (1) modify the WAR file itself, or (2) write `env.properties` to the deployed webapp directory AND make Tomcat not re-deploy from the WAR.

## TheHive 5 and Cortex 4

- **Tailscale for browser access to VM 310**: TheHive 5 has NO init API — first org + admin must be created via web UI wizard. Install Tailscale for browser access via Tailscale IP.
- **TheHive 5 Docker container naming**: Service `thehive` → container `thehive5`. Use `docker logs thehive5` / `docker exec thehive5`.
- **Docker ES vs bare-metal ES port conflict**: Must `systemctl stop elasticsearch` before starting Docker Compose.
- **TheHive 5 Docker images**: TheHive 5 = `strangebee/thehive` (not `thehiveproject/thehive`). Cortex 4 = `thehiveproject/cortex` (not `strangebee/cortex`).
- **TheHive 5 auto-config via env vars**: The entrypoint auto-generates config. BUT: auto-generated cortex config uses `type = "bearer"` with empty key — this DOES NOT WORK because Bearer auth is broken in Cortex. **Must mount a custom `application.conf`** with `type = "basic"`.
- **TheHive 5 first-time default credentials**: login `admin@thehive.local`, password `secret`. The setup wizard appears only when NO users exist.
- **TheHive 5 auto-loads trial platinum license**: On first boot with empty DB, TheHive 5 automatically loads a 15-day trial platinum license. No config option to disable this.
- **TheHive 5 uses Cassandra, not BerkeleyDB**: No data migration from v4. Fresh install only.
- **TheHive 5 requires Elasticsearch for index backend**: The JanusGraph `index.search.backend` MUST be `elasticsearch` (not `lucene`). Lucene causes schema errors. When using Docker entrypoint auto-config, add `--index-backend elasticsearch --es-hostnames elasticsearch` flags.
- **TheHive 5 DB reset procedure**: Drop the Cassandra keyspace AND delete the ES index, then restart TheHive container.
- **Memory**: Docker containers share 6G RAM. Cassandra 512M + ES 512M + Cortex ~512M + TheHive ~1-2G. Monitor with `docker stats`.
- **Cortex 4 Bearer auth is BROKEN**: `MultiAuthSrv` never reaches `KeyAuthSrv`. Bearer API keys are unusable. **Do NOT use `--cortex-keys` or `TH_CORTEX_KEYS`**. Use `type = "basic"` for TheHive→Cortex. For direct Cortex API calls: use CSRF cookie flow or Basic auth.
- **Cortex 4 Basic auth works with config override**: Add `auth.method.basic = true` and `auth.provider = [local, key]` to mounted `/opt/cortex4/conf/application.conf`.
- **Cortex 4 Community Edition analyzer activation**: `POST /api/organization/analyzer/{definitionId}` works for `orgAdmin` via CSRF session auth. **BLOCKING BUG**: Returns 500 due to ES `cortex_6` index mapping conflict. See `references/cortex-analyzer-activation.md` and `references/thehive-cortex-deployment.md`.
- **Cortex 4 CSRF login endpoint is `/api/login`** (NOT `/login`): POST JSON body `{"user":"admin","password":"..."}`. After login, GET `/` to obtain `CORTEX-XSRF-TOKEN` cookie.
- **Cortex 4 CSRF tokens**: Cookie: `CORTEX-XSRF-TOKEN`, header: `X-CORTEX-XSRF-TOKEN`. Flow: login → GET / → extract CSRF cookie → use both cookies + header for POSTs.
- **Cortex 4 init flow**: After fresh start with empty ES index, Cortex enters init mode where org + user creation requires NO auth. After first org+user created, init mode locks permanently. To reset: `curl -X DELETE 'http://localhost:9200/cortex_*'` then `docker restart thehive-cortex`.
- **Cortex 4 user creation `hasPassword: false`**: After creating a user, you MUST set the password separately via `POST /api/user/<login>/password/set` with CSRF session auth.
- **Cortex 4 `relations` ES mapping conflict (BLOCKER)**: The `relations` field stores strings for org/user docs but objects for Worker docs. ES dynamic mapping locks the type based on the first document. This is the current blocker for enabling analyzers programmatically.
- **Cortex 4 CSRF flow (two-step cookie extraction)**: Login only sets a session cookie. You MUST then `GET /` to receive the `CORTEX-XSRF-TOKEN` cookie. Extract and use both cookies + header for all POST operations.
