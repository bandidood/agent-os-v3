# PVE DNAT Rules — Critical ⚠️

When adding DNAT rules on PVE `vmbr0` for port forwarding (e.g., 443→Coolify, 3080→Teleport), the `-d` destination MUST specify the public IP (`95.217.87.114`), NOT `0.0.0.0/0`. Without this, OPNsense's outbound TCP connections to external servers on port 443/80 get intercepted, breaking all internet for VLANs behind OPNsense.

**Correct:**
```
iptables -t nat -A PREROUTING -i vmbr0 -d 95.217.87.114 -p tcp --dport 443 -j DNAT --to-destination 10.0.100.2:443
```
**BROKEN:**
```
iptables -t nat -A PREROUTING -i vmbr0 -p tcp --dport 443 -j DNAT --to-destination 10.0.100.2:443
```

Persist: `iptables-save > /etc/iptables/rules.v4`

Root cause: PVE PREROUTING DNAT matches ALL inbound+transit TCP on vmbr0 with matching dport. OPNsense (10.0.0.2) sends outbound TCP SYN with dport=443 through vmbr0, which matches the broad DNAT and gets hijacked to Coolify instead of reaching the internet.

## OPNsense Outbound NAT

- OPNsense SNAT rules for VLAN→WAN must have `target=wan_ip` (uses OPNsense's WAN interface IP as MASQUERADE source)
- Without SNAT, VLAN clients send packets with their private IP as source → PVE MASQUERADE on vmbr0 handles it, but return traffic DNAT rules can intercept
- After changing iptables rules on PVE, flush conntrack: `conntrack -F`

# Teleport CE on LXC Bastion

- Bastion LXC 308: 10.10.10.108, Teleport v17.5.1 CE
- Ports: 3080 (Proxy/Web), 3023 (SSH), 3025 (Auth), 3024 (Tunnel)
- Config: `/etc/teleport.yaml`
- Teleport .deb pushed via SCP→PVE→`pct push` (no internet initially)
- `tctl users add admin --roles=editor,access` to create admin user
- ACME auto-cert (`teleport.yaml → cluster_name: telep.ccdigital.fr`) fails — HTTP-01 conflicts with Coolify on port 443 (DNAT intercepts port 80 too)
- Use certbot manual DNS-01 challenge for Let's Encrypt:
  1. `certbot certonly --manual --preferred-challenges dns -d telep.ccdigital.fr --email admin@ccdigital.fr --manual-auth-hook /tmp/auth-hook.sh --manual-cleanup-hook "true"`
  2. Auth hook outputs `_acme-challenge.telep.ccdigital.fr TXT <value>`
  3. Add TXT record in Hostinger hPanel
  4. Wait for DNS propagation, then re-run without `--dry-run`
- lego v4.22.0 and acme.sh v3.1.4 installed on bastion — **neither supports Hostinger DNS API natively** (`dns_hostinger` plugin absent in both)
- `acme.sh --dns dns_hostinger` → "Error getting authz" (plugin doesn't exist)
- `lego --dns hostinger` → "unrecognized DNS provider: hostinger"

## Hostinger DNS API

- Hostinger API at `api.hostinger.com/v2` returns **HTTP 403 (error code 1010)** with hPanel tokens
- The token from Hostinger hPanel is NOT the Developer API token — Developer API tokens must be generated separately from hPanel → Developer → API tokens
- **hPanel token ≠ Developer API token** — they are separate auth systems
- For SSL cert automation, use manual DNS-01 challenge until proper Hostinger Developer API token is obtained

## Hostinger DNS TXT Record Naming

When adding `_acme-challenge.telep.ccdigital.fr` TXT record in Hostinger hPanel:
- **Correct Name field**: `_acme-challenge.telep` (Hostinger auto-appends `.ccdigital.fr`)
- **Wrong Name field**: `_acme-challenge.telep.ccdigital.fr` (creates `_acme-challenge.telep.ccdigital.fr.ccdigital.fr` → NXDOMAIN)
- DNS propagation can take 5-15 minutes — verify with `dig +short TXT _acme-challenge.telep.ccdigital.fr @8.8.8.8`

## Teleport Service Configuration

- Config file: `/etc/teleport.yaml` (generated by `teleport configure`)
- Systemd service: `/etc/systemd/system/teleport.service` — `ExecStart=/opt/teleport/system/bin/teleport start -c /etc/teleport.yaml`
- Start: `systemctl daemon-reload && systemctl enable --now teleport`
- Admin user: `pct exec 308 -- tctl users add admin --roles=editor,access`
- Verify: `pct exec 308 -- tctl status` → shows cluster name + version
- After config changes: `systemctl restart teleport`
